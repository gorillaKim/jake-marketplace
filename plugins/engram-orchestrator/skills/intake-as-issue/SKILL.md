# Intake as Issue

## 목적

사용자의 비정형 작업 요청을 곧바로 손대지 말고, **Engram 이슈로 분할·등록한 뒤** 진행하기 위한 게이트.
이슈로 등록하면:
- analyzer 가 작업을 적절히 쪼개고 우선순위와 의존성을 부여
- leader 가 워커에게 분배하고 정체를 감시
- 모든 변경이 issue/task/note 로 추적

## 트리거

다음과 같은 사용자 발화에서 이 스킬을 로드한다:
- "X 를 구현해줘 / 작업해줘 / 추가해줘 / 만들어줘 / 고쳐줘"
- "X 기능 / 모듈 / 컴포넌트 작업"
- 다단계 변경이 명백한 요청 (여러 파일 / 여러 의존 / 새 기능)

다음은 트리거하지 않는다 (직접 처리):
- 단발성 텍스트 편집 ("이 줄 띄어쓰기 고쳐")
- 단순 조회 ("X 가 어디 있어?")
- 명시적 즉시 처리 지시 ("이슈로 만들지 말고 그냥 해줘")

## 실행 절차

### 1) 작업 요청 정리

사용자 요청을 1~3문장으로 재서술해, **무엇을 / 왜** 가 명확한 형태로 정리한다.

### 2) 이슈화 확인

`AskUserQuestion` 으로 사용자에게 확인:

```
질문: "이 작업을 Engram 이슈로 만들어 처리할까요?"
옵션:
  - "예 (Recommended) — analyzer 가 분할/등록 후 작업 시작"
  - "아니오 — 지금 바로 처리"
```

### 3) 분기

- **예**: `Agent(subagent_type='engram-analyzer', prompt=<정리된 요청>)` 호출 →
  analyzer 가 ready 이슈를 생성하면 그 ID 목록을 사용자에게 보고 →
  사용자가 `engram-leader` 를 트리거하거나 즉시 worker spawn 여부 재확인.
- **아니오**: 이슈 생성 없이 즉시 작업. 단 작업 종료 후 사용자에게
  "방금 작업을 회고용 이슈로 등록할까요?" 한 번 더 권한다 (선택).

## 주의

- 사용자가 "이슈로" 라고 명시한 경우는 확인 없이 바로 analyzer 호출.
- analyzer 호출 시 `project_key` 를 함께 전달 — 현재 작업 디렉터리에서 추정하거나, 모르면 한 번 더 사용자에게 질문.
- 이 스킬은 한 세션에서 같은 요청에 두 번 트리거되지 않는다. 한 번 "아니오" 면 그 세션에서는 더 묻지 않음.
